FAQ

COMMON
QUESTIONS.

hello@restoreguard.io

Do I need a Jamf Pro instance to use RestoreGuard?
+
RestoreGuard is designed around Jamf Pro but works independently. Jamf integration unlocks Smart Group targeting, EA sync, and webhook-based auto-enrolment — but the core backup and restore functionality operates without it.
Where is backup data stored?
+
Backup data lives entirely in your infrastructure — we never store it. You choose the destination: encrypted APFS or HFS+ external drives attached to each Mac, your own Amazon S3 bucket, an Azure Blob Storage container, or a Google Cloud Storage bucket. All cloud destinations use your credentials and your account — RestoreGuard orchestrates the transfer but has no access to the stored data. The RestoreGuard server stores metadata only: device health, backup history, and snapshot indexes.
What macOS versions are supported?
+
The RestoreGuard agent requires macOS 13 Ventura or later, and runs natively on both Intel and Apple Silicon. The server runs on any Node.js 18+ environment.
How does the PPPC profile work?
+
We provide a pre-built mobileconfig that grants Full Disk Access via bundleID. No code-signing certificate required for the profile. The PPPC profile must be deployed before the PKG to prevent macOS from caching a denied TCC state.
Is RestoreGuard self-hosted or cloud-hosted?
+
RestoreGuard is currently cloud-hosted only — the management server is fully managed by us, so there's nothing to deploy or maintain on your end. Self-hosted options are on the roadmap for a future release. Note that this refers only to the RestoreGuard management server; your actual backup data always lives in your own infrastructure (external drives or your cloud storage account) and never passes through our servers.

SECURITY

Your data never touches our servers
All backup data is transferred directly from the Mac to your chosen destination — an external drive, your S3 bucket, your Azure container, or your GCS bucket. RestoreGuard's management server handles metadata, scheduling, and reporting only. It never acts as a data relay.
Agent communication
All communication between the RestoreGuard agent and the management server uses HTTPS with TLS 1.2+. API keys are scoped per organisation and can be rotated from the portal at any time. JWT tokens are signed with per-deployment secrets and expire after 24 hours.
Backup encryption
External drive backups use APFS encryption with a key controlled by the device owner. Cloud backups use server-side encryption at the storage provider level (AES-256). Credentials for cloud destinations are stored encrypted and scoped to the minimum required permissions.
Responsible disclosure
If you discover a security vulnerability in RestoreGuard, please report it to security@codefuzion.io. We aim to acknowledge reports within 48 hours and resolve confirmed vulnerabilities within 30 days.

PRIVACY

What we collect. RestoreGuard collects device metadata (serial number, hardware UUID, last backup date, backup health status), admin account information (email address, organisation), and usage events (backup completions, restore initiations). We do not collect, process, or store any backup file content.

What we don't collect. We never access, index, or transmit the contents of backed-up files. Your backup data goes directly to your storage destination. We have no technical access to it.

How metadata is used. Device and backup metadata is used solely to operate the RestoreGuard portal — displaying compliance status, triggering alerts, and generating reports for your organisation. We do not sell metadata to third parties or use it for advertising.

Data retention. Metadata is retained for as long as your account is active. You can delete a device's records from the portal at any time. On account cancellation, all metadata is deleted within 30 days.

Contact. For privacy questions or data deletion requests: privacy@codefuzion.io
